# site-factory — фабрика сайтов для Fiverr (стартовая задача для коворка)

> Этот файл — самодостаточный бриф. Агент работает по нему в **этой папке**.
> Все нужные материалы лежат рядом, в этой же папке (см. раздел «Материалы»).

## Цель проекта

Пайплайн, который **автоматически собирает сайты под заказы Fiverr**: по брифу и референсам
предлагает структуру, собирает сайт из библиотеки секций, деплоит превью на Vercel и сам себя
проверяет. Участие Игоря — только решения (апрув структуры, финальный взгляд), не руки.

Проект обслуживает цель «независимый AI-дизайнер ≥$5k/мес» — фабрика повышает скорость и маржу
гигов. Граница: v1 = один цикл «бриф → превью», не полировать до первого реального заказа.

## Ключевые архитектурные решения (уже приняты, не пересматривать без Игоря)

1. **Code-first.** Источник правды — код (статика HTML/Tailwind → Vercel). Figma — НЕ этап
   пайплайна, а опциональный экспорт-артефакт по требованию клиента.
2. **Структура ≠ скин.** Библиотека делится на:
   - **скелеты секций** (hero, benefits, галерея, отзывы, CTA, футер… — 25–40 шт., Client First-нейминг, слоты под контент);
   - **скины-токены** (палитра, шрифтовая пара, радиусы, тени, плотность отступов).
   «Сайт в стиле референса» = извлечь скин из референса + разложить по своим скелетам.
3. **Figma-экспорт** (когда клиент просит) — через Figma MCP, code-to-design: настоящие
   auto-layout-фреймы, отдельный фрейм на брейкпоинт (1920 / 768 / 375). Не пиксель-в-пиксель:
   видео/анимации/hover упрощаются — это нормально, правда живёт в коде.
4. **Обещание клиенту** — «in the style of your reference», не пиксельный клон (качество + авторские права).

## План этапов

- **Этап 0 — рамка (decide, Игорь):** формат заказа гига (лендинг / многостраничник); подтвердить стек (статика → Vercel; Webflow-ветка позже).
- **Этап 1 — бриф-шаблон:** один md-интейк: ниша, страницы, референсы (обязательное поле, два режима: «мой стиль» = дефолтные токены из портфолио / «стиль референса» = извлечение скина), тексты/ассеты, тон.
- **Этап 2 — структура по референсам:** агент предлагает sitemap + посекционную структуру с обоснованием → апрув Игоря (decide-точка).
- **Этап 3 — сборка:** сайт из библиотеки секций + токены → превью на Vercel.
- **Этап 4 — самопроверка без человека:** скриншоты всех страниц mobile/desktop, битые ссылки, контраст, базовое SEO → отчёт рядом с превью.
- **Этап 5 — накопление:** каждый заказ возвращает новые секции/скины в библиотеку.

## Первый спринт

1. Скелет папки: `README.md` (цель + правила), `backlog.md`, `brief-template.md`, `library/`
   (секции + токены), `orders/` (по папке на заказ), git init.
2. **Извлечь стартовую библиотеку из портфолио:** прогнать доступные сайты по скриншотам,
   из каждого 3–5 секций-скелетов + набор токенов. Нейминг — Client First.
3. Настроить деплой превью на Vercel.

## Тесты приёмки (прогнать после первого спринта)

- [ ] **Тест 1 — сайт из реального референса.** Игорь даёт URL или скриншот живого чужого сайта →
      агент извлекает скин, предлагает структуру, собирает одностраничник из библиотеки,
      деплоит на Vercel. Критерий: «примерно в этом стиле» на глаз Игоря + чек-лист этапа 4 зелёный.
- [ ] **Тест 2 — сайт из Figma-шаблона.** Игорь даёт Figma-файл (шаблон/макет) → агент через
      Figma MCP читает дизайн (design-to-code) и верстает его в код, деплоит на Vercel.
      Критерий: совпадение сетки/типографики/палитры со значениями макета, адаптив 1920/768/375.
- [ ] **Тест 3 — экспорт в Figma.** Готовый сайт из теста 1 → code-to-design в Figma:
      auto-layout-фреймы, 3 брейкпоинта на страницу. Критерий: фреймы реально auto-layout
      (потянул — перестроилось), значения токенов совпадают с кодом.

## Самопроверка (этап 4, гонять на каждом заказе и каждом тесте)

- скриншоты всех страниц на 375 / 768 / 1920;
- битые ссылки и картинки;
- контраст текста (WCAG AA), alt-тексты;
- базовое SEO: title/description, один h1, семантика секций;
- Lighthouse-прогон, если доступен;
- отчёт `orders/<заказ>/selfcheck.md` рядом со ссылкой на превью.

## Решения за Игорем (не делать без него)

- Апрув структуры каждого заказа (этап 2).
- Публикация чего-либо наружу (кроме превью-деплоев на своём Vercel).
- Изменение этого брифа и границ автономии.
